## ADDED Requirements

### Requirement: Finish-options data source
The system SHALL load finish options from a single JSON file at `public/catalog/finish-options.json` and validate it against a Zod schema at load time. The file MUST be generated by a documented seed step (`npm run seed:parts`) from `resources/catalog/部材リスト.xlsx`. If validation fails, the system MUST throw a visible error naming the failing entry and field, and MUST NOT render a partial finish-options UI.

#### Scenario: Valid options file loads successfully
- **WHEN** the app starts and `finish-options.json` passes Zod validation
- **THEN** every option is available to the finish-options UI keyed by `(partId, sheet)`

#### Scenario: Invalid options file blocks startup
- **WHEN** `finish-options.json` fails validation (e.g. an entry has `renderMode` mismatch with its part)
- **THEN** the app surfaces an error naming the failing entry id and field, and does not render the finish-options panel

### Requirement: Finish-option entry shape
Each finish-option entry MUST have a globally unique `id`, a `partId` that resolves to a known part in the active scene's parts manifest, a `sheet` value identifying which workbook sheet it came from (e.g. `"アーバンシー"` or `"レコリード"`), a Japanese `label`, an optional `productCode`, and a `thumbnailUrl` for the swatch image. Each entry MUST set exactly one of `colorHex` (when its part's `renderMode` is `"color"`) or `textureUrl` (when its part's `renderMode` is `"texture"`).

#### Scenario: Color-mode option requires colorHex
- **WHEN** an option's `partId` resolves to a part with `renderMode: "color"` and the option omits `colorHex`
- **THEN** validation fails at load time with an error naming the option id

#### Scenario: Texture-mode option requires textureUrl
- **WHEN** an option's `partId` resolves to a part with `renderMode: "texture"` and the option omits `textureUrl`
- **THEN** validation fails at load time with an error naming the option id

#### Scenario: Setting both colorHex and textureUrl is rejected
- **WHEN** an option declares both `colorHex` and `textureUrl`
- **THEN** validation fails at load time with an error naming the option id

#### Scenario: Duplicate option id rejected
- **WHEN** two options share the same `id`
- **THEN** validation fails at load time with an error naming the duplicated id

### Requirement: Sheet-aware option lookup
The system MUST expose a lookup that returns, for a given `(partId, sheet)`, the ordered list of finish options for that part on that sheet. Sheets MUST be enumerable, and the active sheet MUST be UI-selectable. Switching sheets MUST not silently change the displayed perspective; finish layers continue to render the user's prior selections until the user picks new options for the new sheet.

#### Scenario: Lookup returns options for the active part and sheet
- **WHEN** the active sheet is `"アーバンシー"` and the user selects part `10`
- **THEN** the panel shows every option whose `partId === "10"` and `sheet === "アーバンシー"`, in workbook order

#### Scenario: Empty option set surfaces an empty state
- **WHEN** a `(partId, sheet)` pair has zero options (e.g. ⑮ on a sheet that doesn't expand floor coverage)
- **THEN** the panel shows an empty-state message indicating no options exist for the selected part on the active sheet

### Requirement: Selection preservation across sheet switch
When the active sheet changes, the system MUST preserve any current selection where an option with the same `(partId, label)` exists in the newly active sheet. Selections that do not match MUST be cleared, and the system MUST surface a non-blocking notification that lists the part ids whose selections were cleared and why.

#### Scenario: Matching selection is preserved
- **WHEN** the user has selected part `07` option `"サンドベージュ"` on `"アーバンシー"` and switches to `"レコリード"` where part `07` also has `"サンドベージュ"`
- **THEN** the selection on part `07` remains and the finish layer for part `07` is unchanged

#### Scenario: Unmatched selection is cleared with notification
- **WHEN** the user has a selection on part `15` whose label exists only on `"アーバンシー"` and switches to `"レコリード"`
- **THEN** part `15`'s selection is cleared, its finish layer is removed, and a notification names part `15` as cleared

### Requirement: Explicit "no change" option
For parts where a "no change" / "absent" choice is meaningful (e.g. ② kitchen indirect lighting "光無し", ⑤ range hood "unchanged"), the seed pipeline MUST emit an explicit option entry — `colorHex` set to a transparent-equivalent neutral or `textureUrl` pointing to a transparent PNG sized to the part mask — rather than relying on the absence of any selection. The UI MUST treat this option as a normal chip the user can click.

#### Scenario: Selecting a "no change" option clears the visual finish
- **WHEN** the user selects the explicit "光無し" option on part `02`
- **THEN** part `02`'s finish layer renders with no visible alteration of the base perspective in the masked region
